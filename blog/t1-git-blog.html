Version Control, Git and Why You Need It

A Beginner's Guide

25 January 2015

Learning to code involves more than learning the syntax of your language(s) of choice. There are some important basics to know that are commonly thought of as good practice within coding. I’ll call these the Recommended Coding Guidelines. The first, and most important, of these is version control.

What is version control and what are the benefits of it?

Often when you work on something, let’s say a document in Microsoft Word, you make alterations to your file and then save it, thereby overwriting the previous content of the document. Later, you might decide you regret deleting a particular paragraph. If you haven’t turned on track changes you’ll probably panic. However, if you’ve savvily turned them on Word will let you see that deleted paragraph and, loosely speaking, reinstate it from the past version of your document to your current version. This is like the distant cousin of version control.

In coding, if you worked on a piece of code and then saved it, overwriting the previous content, then discovered that you’d made an error that, unless you had a method of tracking your changes you’d have to undo all you work manually to resolve your problem. That could take a very long time. Version control helps manage this.

Version control lets one person or many people make changes to a project. These changes are saved at different points but each save does not overwrite the previous incarnation of the project. This means, if necessary, you can see and go back to those previous states.

A popular and free open source method of version control is Git. You should learn it.

How does git help you keep track of changes?

Each new project you use Git for works like a tree. Just like a tree Git lets you have branches. The master is the main copy of you project repository. If you want to make changes you can create a branch off the master to try out the changes before saving them back to your main project. If you are feeling experimental whilst making that change you can branch off it, too. Each branch creates a new version and can be merged back into the master in order to include those changes or experimentations in it. You don’t have to keep track of enumerable branches and changes in your head, Git does this for you.

Why use GitHub to store your code?

There are different types of version control. You don’t need to know how they all work, though more about them can be found here. You do need to know about distributed version control systems. They mirror versions stored locally on your machine on a separate server. This way everything is fully backed up if anything happens to any machine within the system.

Git works locally on your machine, letting you branch off your master, creating different versions and, generally, not do terrible things by overwriting your very fine work. But in order to be doubly safe you’ll want to back that up offsite, and if you want to work with others on the project you’ll need somewhere they can access your Git repository. GitHub lets you do this. It is the hub of git and lives online.
